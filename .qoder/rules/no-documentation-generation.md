# 禁止生成说明文档规则

## 规则概述

在项目开发过程中，严格禁止生成任何 Markdown **说明性文档**文件。所有使用说明、实现总结、指南等内容必须直接在聊天框中输出，供用户实时查看。代码文件、配置文件、规则文件、Prompt 模板等功能性文件正常创建和修改。

---

## 核心要求

### 1. 禁止生成的文档类型
- ❌ README.md、QUICK_START.md、USAGE.md 等使用指南文档
- ❌ IMPLEMENTATION_SUMMARY.md、CHANGELOG.md 等总结性文档
- ❌ GUIDE.md、TUTORIAL.md 等教程文档
- ❌ API.md、SPECIFICATION.md 等规范文档
- ❌ 任何以 `.md` 为扩展名的**说明性**文件
- ⚠️ 注意：规则文件（.qoder/rules/*.md）、prompt 模板等功能性文件除外

### 2. 允许创建的文件类型
- ✅ 用户明确要求创建的任何文件
- ✅ 代码文件（.java、.js、.ts、.py 等）
- ✅ 配置文件（pom.xml、application.yaml、package.json 等）
- ✅ 规则文件（.qoder/rules/*.md）
- ✅ Prompt 模板文件（resources/prompt/*.md、*.txt）
- ✅ 代码注释和 JavaDoc 文档
- ✅ 项目正常开发所需的任何非说明性文件

### 3. 输出方式要求

**必须做（MUST DO）：**
- 所有实现说明、使用指南直接在聊天框中输出
- 使用 Markdown 格式在聊天中展示代码示例
- 通过聊天消息提供分步骤的操作指导
- 在聊天中总结实现要点和注意事项
- 正常创建和修改代码、配置、规则等必要文件

**禁止做（MUST NOT DO）：**
- 不要使用 `create_file` 工具创建**说明性** `.md` 文档（如 QUICK_START.md、SUMMARY.md 等）
- 不要说"我将创建一个文档来说明使用方法..."
- 不要在实现功能后自动生成总结文档
- 不要创建多个说明文档来组织内容
- 不要为了记录实现过程而创建文档

---

## 执行标准

### 当完成一个功能实现时：

**❌ 错误做法：**
```
我已经完成了功能实现，现在我将创建以下文档：
1. QUICK_START.md - 快速开始指南
2. IMPLEMENTATION_SUMMARY.md - 实现总结
3. API_GUIDE.md - API 使用指南
```

**✅ 正确做法：**
```markdown
✅ 功能实现已完成！以下是使用说明：

## 快速开始

1. 启动服务：
   ```bash
   mvn spring-boot:run
   ```

2. 调用接口：
   ```bash
   POST /api/example
   {
     "param": "value"
   }
   ```

## 实现要点

- 使用了 Spring Boot 3.4.5
- 集成了 Redis 缓存
- 支持事务管理

## 注意事项

- 确保 Redis 服务已启动
- 需要配置数据库连接
```

### 当用户询问如何使用时：

**✅ 直接在聊天中提供完整说明：**
- 分步骤列出操作流程
- 提供代码示例和配置示例
- 说明关键参数和注意事项
- 给出常见问题的解决方案

---

## 适用场景

此规则适用于：
- ✅ 功能实现后的说明
- ✅ 使用指南的提供
- ✅ 实现总结的输出
- ✅ API 文档的展示
- ✅ 配置说明的描述
- ✅ 问题排查指导

---

## 例外情况

以下情况可以创建 Markdown 文件：
1. **用户明确要求**：用户直接说"创建一个 README.md"或"生成文档"
2. **功能性文件**：
   - 规则文件：`.qoder/rules/*.md`
   - Prompt 模板：`resources/prompt/*.md`
   - 知识库文件：作为 AI 输入的知识文件
3. **项目初始化**：脚手架生成的默认文档
4. **所有代码和配置文件**：正常的项目开发文件

---

## 违规处理

如果发现自己正在创建**说明性文档**文件，应：
1. 立即停止文档创建
2. 将说明内容改为聊天输出
3. 向用户解释已在聊天中提供相关信息
4. 继续正常创建代码、配置等必要文件

---

## 最佳实践

### 聊天输出的组织方式：

```markdown
# 功能名称

## 概述
简要说明功能目的

## 使用步骤
1. 第一步
2. 第二步
3. 第三步

## 代码示例
```java
// 示例代码
```

## 配置说明
- 配置项1: 说明
- 配置项2: 说明

## 注意事项
- 注意点1
- 注意点2
```

---

## 优先级

此规则的优先级为 **最高级**，在与其他项目规范冲突时：
- ⭐ 优先执行此规则（不生成说明文档）
- ⭐ 通过聊天补充规范要求的说明内容
- ⭐ 仅在用户明确要求时例外
- ⭐ 代码、配置、规则等文件正常创建

---

**总结：从现在开始，所有说明都通过聊天直接展示，不再创建说明性 Markdown 文档文件，但代码、配置、规则等必要文件正常创建！** 🎉
